# MDWA product page

A new product information page at `/pages/mdwa.html` introduces Manulife Dynamic Wealth Assurance into the site's product catalogue. The page follows the established pattern from `mdla.html` and `mps-flexi.html`: a hero section, plan comparison cards, advantages, eligibility criteria, a static premium table, an interactive calculator, and a CTA footer. Content is written in question-style language ("bahasa pertanyaan") and the calculator uses JSON data lookup with no formulas. The `index.html` already carries a pre-existing link to `/pages/mdwa.html`, so the page connects to the catalogue without any routing changes.

**Watch for:** The rider ALP+ JSON data is drawn entirely from the FEAT-001 task specification, not the source PDFs/spreadsheets — it couldn't be verified against `Sample/Mdwa.html`, which contains no rider pricing rows (confirmed). Any numeric error in the task spec would silently propagate to the live page. Also, the calculator is scoped to Plan A IDR sekaligus only, but a user selecting 100 Juta will see "Data tidak tersedia" for the rider line with no explanation at the point of interaction, only in the footnote below the result box.

---

## High-level view

The page structure and visual system are a faithful port from the existing product pages. All shared CSS variables, the component-loader pattern (`site-header`/`site-footer` divs + `main.js`), dark mode, scroll animations, and WhatsApp CTA links are applied consistently. The static premium table values match `Sample/Mdwa.html` exactly across all twelve rows.

The rider data gap is the main functional concern. The calculator's JSON embeds rider ALP+ entries for `Manfaat_50_Juta` only. Selecting 100 Juta correctly falls through to "Data tidak tersedia" for the rider row, and the total shows a `(tanpa rider)` suffix. The footnote explains this, but it sits below the result box — a user who sees a partial result may interpret the rider absence as a bug rather than a data limitation.

The page uses `!important` on button and section-tag overrides where the existing product pages achieve the same colour customisation via class specificity alone, creating a minor maintenance divergence.

### Rider data provenance

The embedded `mdwaData` object contains two base-benefit lookups (50 Juta and 100 Juta, each for 20 and 30 year terms) and two rider blocks for ALP+ (50 Juta × 20 years, 50 Juta × 30 years). The base-benefit values — 527,500,000 / 725,000,000 / 1,055,000,000 / 1,440,000,000 — match the source `Sample/Mdwa.html` tables exactly. The rider values cannot be cross-referenced within the repo: `Sample/Mdwa.html` lists ALP Plus as a rider by name but contains no pricing table for it, and the source Excel files (`Kalkulator Profit MDWA B (IDR).xlsx`) are binaries that can't be read here. The rider numbers come exclusively from the FEAT-001 task spec. If those numbers were transcribed incorrectly, the calculator will show wrong premiums without any runtime signal.

### UX asymmetry when selecting 100 Juta

When the user selects Besar Manfaat = 100 Juta, `calcMdwa()` constructs the key `Manfaat_100_Juta_20_Tahun_Sekaligus` for the rider lookup, which has no entry in the rider JSON. The rider line shows "Data tidak tersedia" and the total reads the base premium plus `' (tanpa rider)'`. The explanation — "Data rider ALP+ hanya tersedia untuk manfaat 50 Juta" — is in a `<p>` footnote rendered *after* the result box. A user on mobile who doesn't scroll past the result will see a partial result with no explanation. Surfacing the scope note inline (e.g. a muted label under the 100 Juta option button, or conditionally hiding the rider row when 100 Juta is selected) would prevent misreading.

### `!important` in page-specific overrides

`.mdwa-btn-primary`, `.mdwa-btn-outline`, and `.mdwa-section-tag` all declare `!important` on colour and background properties to win specificity against the shared `.btn` and `.section-tag` rules in `style.css`. This means the shared button styles can never affect these components on the MDWA page without adding more `!important` in the shared sheet — a specificity ratchet that makes global theming harder. The other product pages use custom class names without `!important`, suggesting the same approach is available here.
