# MiPrecious product page with premium calculator

A new 1000-line product page for MiPrecious (MiPreparation Legacy for Our Assurance), an endowment life insurance product from Manulife. The page follows the project's established pattern from msp.html: shared style.css, dark-mode init script, page-scoped CSS variables, component loader stubs (`site-header`, `site-footer`), and `main.js` at the end. Content is presented in question-style Indonesian ("Pernahkah Anda...?", "Tahukah Anda...?") with colored keyword spans. The calculator computes guaranteed benefits (10% annual + 25% decadal + 100% maturity) and projected additional benefits from surplus.

**Watch for:** The calculator's guaranteed-benefit formula produces 585% (50yr) and 355% (30yr), contradicting the 550%/320% figures stated in the page's own marketing copy and benefit cards (confirmed). The mode-payment selector remains active when "Sekaligus" payment is chosen, producing nonsensical output labels like "Premi Sekaligus (Bulanan)" (confirmed).

## High-level view

The page structurally mirrors msp.html with product-appropriate color theming (`--mip-*` palette), additional keyword color classes, and the same section anatomy. Design system integration consumes CSS custom properties from style.css throughout, dark mode toggles via `html.dark` (21 rules), and responsive grids use `auto-fit`/`minmax`. The page redefines `.kw-gold` (#F57F17) over the global style.css definition (#B8860B), which works via source order but creates a maintenance fragility.

The calculator takes Dana Masa Depan as input, derives Premi Sekaligus via the DMD/1.70 ratio, applies modal factors for sub-annual payment modes, classifies the plan tier, and renders guaranteed + projected benefit breakdowns. The core benefit-counting loop correctly excludes decadal years from the annual tally, but the resulting sum (585%/355%) does not match the marketed totals (550%/320%) that appear in the surrounding page content.

The "Sekaligus" payment mode logically implies a single lump-sum payment, yet the UI allows selecting sub-annual modes (Semesteran, Kuartalan, Bulanan) and multiplies the lump sum by the modal factor, producing a fractional figure that has no real-world meaning for a single-premium product.

## Benefit calculation discrepancy

The `calculateMIP` function counts annual payments from year 11 to `term - 1` (inclusive), skipping multiples of 10:

```javascript
for (var yr = 11; yr <= term - 1; yr++) {
  if (yr % 10 !== 0) numAnnual++;
}
```

For a 50-year term, this yields years 11-19 (9), 21-29 (9), 31-39 (9), 41-49 (9) = 36 annual payments at 10% = 360%. Decadal payments: `Math.floor(50/10) = 5` at 25% = 125%. End-of-term: 100%. Grand total: 585%.

The benefit card section of the same page states:

> **50 tahun:** 5 kali pembayaran = 125% DMD (decadal)
> **50 tahun:** 36 kali pembayaran = 360% DMD (annual)
> **100% x DMD** (end)
> Total Manfaat Pasti: **550%** (50thn)

125 + 360 + 100 = 585, not 550. The Sample/Miprecious.html reference has the identical logic in its calculator and the same 550% marketing claim, so this discrepancy originates in the source material. However, the Sample's illustration table shows year 50 receiving BOTH the sepuluh-tahunan (25%) AND the akhir masa (100%), which confirms the calculator's 585% is the actual mathematical total.

If the intended logic is that the year-50 decadal payment is subsumed by the maturity payment (only 100% paid at year 50, not 100% + 25%), then `numDecadal` should be `Math.floor(term/10) - 1` (4 for 50yr, 2 for 30yr), giving 100% + 360% + 100% = 560% for 50yr. That still doesn't equal 550%, suggesting there may be one fewer annual payment in the official product rules. Without the official product illustration PDF being machine-readable, the exact correction cannot be confirmed.

**Recommendation:** Cross-reference with the official Manulife illustration (Source/MiPrecious/C1. Ilustrasi MiPrecious 300jt...) to determine the correct counting rule, then fix either the calculator logic or the marketing copy.

## Mode selector interaction with Sekaligus payment

When `mipState.payment === 'Sekaligus'`, the code executes:

```javascript
masaBayarTahun = 1;
var modalFactor = MIP_MODAL_FACTORS[cur][mode];
premiPerPeriod = premiSekaligus * modalFactor;
premiLabel = 'Premi Sekaligus (' + mode + ')';
```

If a user selects Sekaligus + Bulanan (IDR), the displayed "Premi Sekaligus (Bulanan)" = `premiSekaligus * 0.095`. For a DMD of 100M (premi ~58.8M), this shows ~5.6M labeled as if it's a single premium payment in "Bulanan" mode. The concept of paying a single premium in monthly installments contradicts the "Sekaligus" label.

The `totalPremi` variable is always set to `premiSekaligus` regardless of mode, so the comparison section ("Perbandingan Premi vs Manfaat") remains correct. The confusion is limited to the "Informasi Premi" display rows, but a user seeing "Premi Sekaligus (Bulanan) = Rp 5.588.235" could misunderstand the actual cost of the product.
