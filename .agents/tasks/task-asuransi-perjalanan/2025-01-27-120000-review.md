# Travel Insurance product page with premium calculator

A new static HTML product page for travel insurance (`asuransi-perjalanan.html`) that mirrors the structural pattern of `asuransi-mobil.html`: hero section, benefit cards, region/plan comparison, a premium table, an interactive calculator covering 3 age groups x 3 coverage types x 5 regions x 2 plans x 8 duration brackets (plus a >31-day overflow formula), FAQ accordion, and CTA. The page uses a teal/green color palette, question-style copywriting throughout, inline dark-mode support, and embeds the full premium lookup table directly in the page's JavaScript.

**Watch for:** The 70-84y age group UI lets users select Deluxe combinations that are `null` in the data — the plan dropdown disables only for Schengen Basic but not for 70-84y where Deluxe is universally unavailable (**confirmed**). The premium data table is ~120 lines of raw numbers embedded in HTML with no source-of-truth reference, making future updates error-prone (**confirmed**). The `toggleFaq` function is global-scoped, creating a collision risk if multiple pages are loaded as fragments or the function name is later moved into `main.js` (**likely**).

## High-level view

The page follows the existing product-page template exactly: inline `<style>` with custom CSS vars and keyword highlight classes, semantic sections loaded into a shared shell via `main.js`, and a calculator wrapped in an IIFE. The structure is a straight adaptation — different color palette, different content, same skeleton.

The premium calculator maps a 4-dimensional input space (age, coverage, region, plan) to a rates array and a per-week overflow value. Null entries (Schengen Basic for 70-84y, Deluxe for 70-84y, Deluxe for Schengen Basic) produce user-facing error messages rather than crashes. However, the UI only proactively prevents impossible selections for the Schengen Basic constraint, not for the 70-84y age group where Deluxe is equally invalid.

The embedded premium data is large and hardcoded with no provenance annotations. Future rate changes will require hand-editing ~120 lines of nested objects inside an HTML file with no way to verify against the source document.

## Calculator null-path asymmetry

The `handleRegionChange` function disables the plan dropdown when the user selects Schengen Basic, preventing selection of the unavailable Deluxe plan. The same treatment isn't applied to the age group dimension. When a user selects "70 - 84 tahun", every Deluxe entry in the data is `null`, yet the plan dropdown stays enabled. The user can pick Deluxe, click calculate, and get an error message ("Opsi tidak ditanggung") — functional but inconsistent with the Schengen Basic behavior.

The fix: add a listener on `#travel-age` that mirrors `handleRegionChange`:

```javascript
function handleAgeChange() {
  var age = document.getElementById('travel-age').value;
  var planSelect = document.getElementById('travel-plan');
  if (age === '70_84y') {
    planSelect.value = 'superior';
    planSelect.disabled = true;
  } else if (document.getElementById('travel-region').value !== 'schengen_basic') {
    planSelect.disabled = false;
  }
}
```

The compound condition (age AND region) needs care — if 70-84y is selected AND Schengen Basic is selected, the plan should remain disabled regardless of which constraint relaxes first. A single `updatePlanState()` helper called from both change handlers would be cleaner than two independent functions.

## Inline premium data without provenance

The `premiumData` object is a 120-line nested structure containing rates for 3 x 3 x 5 x 2 = 90 plan-rate-arrays plus per_week values. There's no comment indicating which document these numbers were transcribed from, making it impossible for a future maintainer to verify correctness or apply rate updates without reverse-engineering the source. At minimum, a comment like `// Source: Travel Insurance Rate Table v2.3 (Jan 2025)` at the top of the data block would anchor the values to a verifiable document.

## Premium table vs calculator data drift risk

The rendered HTML premium table and the JS `premiumData` object encode the same rates independently. Since the table is hand-written HTML and the calculator data is a separate JS object, a drift could emerge if only one is updated during a rate change. Both need to be updated in lockstep, and there's no mechanism enforcing that today.
